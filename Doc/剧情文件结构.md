# 剧情文件结构

## 说明

本文基于以下信息整理：

- `D:\codes\LongYin\Meta\dump.cs` 中的类与枚举定义
- `Assembly-CSharp` 的 method dump
- 导出的 `Assets\Scripts\Assembly-CSharp` / `Assets\Scenes` 反编译结果

本文只聚焦两类“剧情配置核心对象”：

1. `MissionData`
2. `WorldPlotEventData`

其中：

- `MissionData` 是“任务模板数据”。
- `WorldPlotEventData` 是“世界剧情触发规则数据”。
- 两者都不是最终对白文本本体。
- 真正的对白内容主要仍然落在 `PlotData` 上，`WorldPlotEventData` 则主要通过 `plotID` 去引用它。

额外说明：

- 文中凡是标注“推断”的地方，表示该解释来自字段名与调用方式，不是源码中直接写出的中文注释。
- JSON 示例使用 `jsonc` 语法，方便直接在结构里写注释。
- 类里存在但不会进入实例 JSON 的 `static` 字段，会单独说明。
- 一些字段属于运行时状态，而不是策划静态配置；这类字段也会明确标出。

## 1. MissionData

### 1.1 类的作用

`MissionData` 是任务系统里的“任务模板”。  
任务控制器中保存的是模板列表，例如：

- `MissionDataController.MainMissionDataBase`
- `MissionDataController.BranchMissionDataBase`
- `MissionDataController.LittleMissionDataBase`
- `MissionDataController.bountyMissionDataBase`
- `MissionDataController.SpeKillerMissionDataBase`
- `MissionDataController.TreasureMapMissionDataBase`

在运行时，`PlotController.GetMainMission(int)`、`PlotController.GetBranchMission(int)` 这类方法会先从这些列表里取出一条 `MissionData`，再 `Clone()` 一份交给当前存档使用。  
因此它本质上是：

- “任务原型”
- “任务发放前的配置数据”
- “任务说明、目标、触发点、奖励、附带剧情事件的总入口”

可以把它理解为：

```text
MissionData = 一个任务条目的完整模板
MissionTargetData = 这个任务中的一个目标阶段
MissionNeedData = 这个目标阶段里的一个计数/达成条件
EventData = 任务在地图上生成的目标事件
PlotData = 事件里真正播放的剧情内容
```

### 1.2 MissionData 主结构

下面是按 JSON 形式写出的主结构。  
为了可读性，嵌套对象只保留骨架，详细说明放在后文。

```jsonc
{
  "name": "初习武艺", // 任务名
  "speMissionID": -1, // 特殊任务编号，-1 通常表示无
  "leftTime": 30, // 总剩余天数
  "stageMinLeftTime": 0, // 阶段最少剩余天数
  "difficulty": 3.0, // 任务难度
  "difficultyRate": 1.0, // 难度倍率
  "rewardRate": 1.0, // 奖励倍率
  "treasureLv": 0.0, // 藏宝/赏金类任务常用的宝物等级参数
  "bountyTypes": [0, 2], // 可用悬赏类别，见 BountyType
  "minForceLv": 0, // 最低势力等级要求
  "missionSourceType": 0, // 任务来源类型，见 MissionSourceType
  "missionBountyType": 0, // 当前悬赏类型，见 BountyType
  "sourceHeroID": -1, // 任务发布者/来源角色 ID
  "sourceForceID": 0, // 任务来源势力 ID
  "missionHeroID": -1, // 任务核心关联角色 ID
  "missionDisableQuickTravel": false, // 任务期间禁用快速移动
  "missionHideTargetPlace": false, // 是否隐藏目标地点
  "missionHideTargetPlaceString": "", // 运行时写入的隐藏地点描述文本
  "missionRandomSeed": 0, // 运行时随机种子
  "missionTargetDatas": [
    {
      "describe": "前往#TargetPlace#与某人交谈", // 目标说明模板
      "missionEventDataSaveRecord": 0, // 运行时事件存档记录位
      "missionEventData": {
        "...": "见 1.5 EventData"
      },
      "missionTriggerType": 5, // 见 MissionTriggerType
      "missionTargetAreaTypeLimit": 0, // 见 MissionTargetAreaTypeLimit
      "tirggerTargetID": "姜映泉", // 注意：原字段名就是 tirggerTargetID，游戏里拼写如此
      "missionNeedDatas": [
        {
          "missionTargetType": 3, // 见 MissionTargetType
          "missionTargetID": "姜映泉",
          "missionResourceNeed": 0,
          "missionNumCount": 0.0, // 运行时当前进度
          "missionNumNeed": 1.0 // 目标值
        }
      ],
      "missionRequirementTypeList": [0, 25], // 见 ChoiceRequirementType
      "missionRequirementType": 0, // 单项要求写法，和上面的列表并存
      "missionRequirementNum": 20.0,
      "missionTargetFinishCallPlotID": -1 // 该目标完成后调用的剧情 ID
    }
  ],
  "noAutoFinish": false, // 是否禁止自动结算
  "missionFunds": 0, // 任务经费
  "missionFameReward": 0.0, // 名望奖励
  "missionContributionRewardBase": 1.0, // 基础贡献系数
  "missionContributionReward": 0.6, // 实际贡献奖励
  "missionMoneyReward": 500, // 银两奖励
  "missionJoinTeamHero": "甲/乙", // 接任务时自动入队的角色名列表，/ 分隔
  "missionJoinTeamHeroID": [1001, 1002] // 运行时解析后的入队角色 ID 列表
}
```

### 1.3 MissionData 字段说明

#### 根字段

- `name`
  - 任务名，也是复杂 JSON 补丁按 `name` 匹配覆盖时使用的主键。

- `speMissionID`
  - 特殊任务编号。
  - 常见默认值是 `-1`，表示没有额外特殊编号。

- `leftTime`
  - 任务总剩余时间，单位通常是“天”。

- `stageMinLeftTime`
  - 阶段最少剩余时间。
  - 按命名看，它更像阶段切换时的下限控制字段。

- `difficulty`
  - 任务当前难度值。
  - 任务描述、生成内容、奖励强度会围绕它变化。

- `difficultyRate`
  - 难度倍率。

- `rewardRate`
  - 奖励倍率。

- `treasureLv`
  - 藏宝图、悬赏、搜刮类任务会用到的宝物等级/宝藏层级参数。
  - 这不是传统意义上的“任务等级”。

- `bountyTypes`
  - 该模板允许被哪些悬赏类型复用。
  - 常用于悬赏任务模板池。

- `minForceLv`
  - 任务要求的最低势力等级。

- `missionSourceType`
  - 任务所属的大类来源。
  - 主线、支线、势力任务、悬赏任务都在这里区分。

- `missionBountyType`
  - 当前这条任务实例实际采用的悬赏类别。

- `sourceHeroID`
  - 任务发布人、来信人、任务来源角色。
  - `MissionData.GetMissionDescribe()` 一类逻辑会用它替换说明文本中的角色代称。

- `sourceForceID`
  - 任务来源势力 ID。

- `missionHeroID`
  - 任务核心关联角色 ID。
  - 常见于“找某人”“护送某人”“击败某人”这类任务。

- `missionDisableQuickTravel`
  - 任务进行中是否禁用快速移动。

- `missionHideTargetPlace`
  - 是否隐藏目标地点。
  - 打开后，运行时会把模糊地点描述写入 `missionHideTargetPlaceString`。

- `missionHideTargetPlaceString`
  - 运行时字段。
  - `GameController.GetFullMission()` 中会在任务正式发放时填入“模糊地点描述”。

- `missionRandomSeed`
  - 运行时字段。
  - 任务被正式接取时写入随机种子，用来固定本任务的随机内容。

- `missionTargetDatas`
  - 任务目标阶段列表。
  - 一条任务可以由多个目标阶段组成。

- `noAutoFinish`
  - 是否禁止自动完成。
  - 势力任务生成逻辑里会主动把它设为 `true`。

- `missionFunds`
  - 任务经费。
  - UI 和运行时逻辑里都会直接显示/扣取这笔资金。

- `missionFameReward`
  - 名望奖励。

- `missionContributionRewardBase`
  - 基础贡献奖励系数。
  - `SetDifficulty()` 会根据它和 `difficulty` 重新计算 `missionContributionReward`。

- `missionContributionReward`
  - 当前实际的势力贡献奖励。

- `missionMoneyReward`
  - 银两奖励。

- `missionJoinTeamHero`
  - 字符串形式的“接任务后自动入队角色名列表”。
  - `GameController.GetFullMission()` 会按 `/` 分割，然后让这些角色直接加入队伍。

- `missionJoinTeamHeroID`
  - 运行时字段。
  - 上面的 `missionJoinTeamHero` 在接任务时被解析成 ID 后，会写进这里。

#### 类中存在但不会进入实例 JSON 的静态字段

- `MissionBountyTypeRewardType`
  - 赏金任务奖励类型辅助表。

- `MissionTargetTypeBaseMissionContribution`
  - 不同任务目标类型的基础贡献系数表。

- `MissionTargetNpc`
  - 哪些 `MissionTargetType` 被任务系统视为“NPC 相关目标”的辅助列表。

#### MissionData 中常见的运行时回填字段

这些字段虽然属于类结构的一部分，但更接近“任务实例状态”而不是“原始策划配置”：

- `missionHideTargetPlaceString`
  - 接任务时根据真实目标地点动态生成。

- `missionRandomSeed`
  - 接任务时生成。

- `missionJoinTeamHeroID`
  - `missionJoinTeamHero` 被解析后写入。

- `missionTargetDatas[].missionEventDataSaveRecord`
  - 存档记录位。

- `missionTargetDatas[].missionNeedDatas[].missionNumCount`
  - 当前完成进度。

- `missionTargetDatas[].missionEventData.seen`
- `missionTargetDatas[].missionEventData.happened`
- `missionTargetDatas[].missionEventData.noticed`
- `missionTargetDatas[].missionEventData.hovered`
- `missionTargetDatas[].missionEventData.randomSeed`
  - 都属于事件运行时状态。

### 1.4 MissionTargetData

`MissionTargetData` 表示一个任务阶段或一个任务目标。

```jsonc
{
  "describe": "前往#TargetPlace#交谈",
  "missionEventDataSaveRecord": 0,
  "missionEventData": { "...": "见 EventData" },
  "missionTriggerType": 5,
  "missionTargetAreaTypeLimit": 0,
  "tirggerTargetID": "姜映泉",
  "missionNeedDatas": [ { "...": "见 MissionNeedData" } ],
  "missionRequirementTypeList": [0, 25],
  "missionRequirementType": 0,
  "missionRequirementNum": 20.0,
  "missionTargetFinishCallPlotID": -1
}
```

- `describe`
  - 阶段说明模板。
  - 常见占位符如 `#TargetPlace#`、`#NextTargetPlace#` 会由运行时替换。

- `missionEventDataSaveRecord`
  - 运行时事件记录位。
  - 用于把该目标对应的地图事件、事件状态写进存档。

- `missionEventData`
  - 该目标生成在地图上的事件数据。
  - 比如“去某处调查”“与某人对话”“打一个随机遭遇”都要靠它落到地图上。

- `missionTriggerType`
  - 玩家做什么行为会触发或推进这个目标。

- `missionTargetAreaTypeLimit`
  - 目标地点生成时的区域限制。

- `tirggerTargetID`
  - 触发目标 ID。
  - 字段名有原版拼写错误：`tirgger` 而不是 `trigger`。
  - 其含义取决于 `missionTriggerType`，可能是区域 ID、建筑键、角色名、技能 ID、书籍 ID 等。

- `missionNeedDatas`
  - 该目标的完成条件列表。
  - 计数、收集、好感、武学、探索等都在这里定义。

- `missionRequirementTypeList`
  - 额外门槛列表。
  - 类型使用 `ChoiceRequirementType`，也就是和剧情选项共用的那套“属性/技能要求”枚举。

- `missionRequirementType`
  - 单项额外门槛写法。
  - 与上面的列表同时存在，说明系统同时兼容“单条件”和“多条件”两种写法。

- `missionRequirementNum`
  - 额外门槛需要达到的数值。

- `missionTargetFinishCallPlotID`
  - 当前阶段完成后自动调用的剧情 ID。
  - 默认 `-1` 表示不调用。
  - 这是任务阶段推进到剧情桥段的关键字段之一。

### 1.5 MissionNeedData

`MissionNeedData` 是一个具体的完成条件。

```jsonc
{
  "missionTargetType": 3,
  "missionTargetID": "姜映泉",
  "missionResourceNeed": 0,
  "missionNumCount": 0.0,
  "missionNumNeed": 1.0
}
```

- `missionTargetType`
  - 条件类型。
  - 比如探索地区、提升好感、招募角色、学习新技能、收集宝物等。

- `missionTargetID`
  - 条件目标 ID。
  - 类型是字符串，是因为有时存的是纯数字 ID，有时存的是名字或复合键。

- `missionResourceNeed`
  - 需要的资源类型编号。
  - 多用于“交某种资源”“消耗某类物资”。

- `missionNumCount`
  - 当前进度。
  - 这是明显的运行时字段。

- `missionNumNeed`
  - 目标值。

### 1.6 EventData

`EventData` 是地图事件对象。  
在任务系统里，它负责把“任务目标”变成玩家在大地图/区域地图上能看到或接触到的实际事件点。

```jsonc
{
  "eventName": "遭遇邪道",
  "eventDescribe": "在#PosText#一带似乎有人埋伏。",
  "eventAvailableAreaType": [0, 2],
  "spriteName": "event_icon",
  "isAreaEvent": true,
  "resourcePointID": 0,
  "areaID": [55],
  "areaMapTileID": [],
  "bigMapPos": { "x": 123.0, "y": 456.0 },
  "nearAreaID": 55,
  "nearAreaDirection": 1,
  "seen": false,
  "happened": false,
  "noticed": false,
  "hovered": false,
  "plotTargetEvent": false,
  "missionTargetEvent": true,
  "autoDestroy": true,
  "leftTime": 10,
  "difficulty": 3.0,
  "difficultyRate": 1.0,
  "speTargetID": 0,
  "plotData": { "...": "见 PlotData" },
  "eventItemList": { "...": "见 ItemListData" },
  "leaderLimit": 0,
  "eventOutTimeCallFuc": "",
  "seeRange": 0.0,
  "randomSeed": 0,
  "inaccuracyPosText": false,
  "notImportant": false
}
```

- `eventName`
  - 事件名。

- `eventDescribe`
  - 事件描述文本。

- `eventAvailableAreaType`
  - 该事件允许出现在哪类地图区域。

- `spriteName`
  - 事件图标。

- `isAreaEvent`
  - 是否是区域内事件。
  - `false` 时往往更偏向大地图事件。

- `resourcePointID`
  - 资源点 ID。
  - 资源点型事件会用到。

- `areaID`
  - 允许落点的区域 ID 列表。

- `areaMapTileID`
  - 更细粒度的区域 tile 列表。

- `bigMapPos`
  - 大地图坐标。

- `nearAreaID`
  - 邻近区域 ID。

- `nearAreaDirection`
  - 邻近方向。

- `seen` / `happened` / `noticed` / `hovered`
  - 运行时状态字段。
  - 已看见、已发生、已注意、已悬停这类 UI/状态标记。

- `plotTargetEvent`
  - 是否为剧情系统生成的目标事件。

- `missionTargetEvent`
  - 是否为任务系统生成的目标事件。

- `autoDestroy`
  - 事件触发后是否自动销毁。

- `leftTime`
  - 事件剩余存在时间。

- `difficulty`
  - 事件难度。

- `difficultyRate`
  - 事件难度倍率。

- `speTargetID`
  - 特殊目标 ID。
  - 常用于事件内部的特例记录。

- `plotData`
  - 事件附带的剧情对象。
  - 这已经是加载后的 `PlotData` 运行时结构，而不是原始 CSV 文本。

- `eventItemList`
  - 事件附带的物品列表。

- `leaderLimit`
  - 是否要求队长触发。

- `eventOutTimeCallFuc`
  - 事件超时后执行的特殊函数。

- `seeRange`
  - 事件可见范围。

- `randomSeed`
  - 事件随机种子。

- `inaccuracyPosText`
  - 是否使用模糊位置文本。

- `notImportant`
  - 是否被视为“不重要事件”。

### 1.7 PlotData

`PlotData` 是真正的剧情内容对象。  
它和 `PlotData.csv` 的关系是：

- `PlotData.csv` 是原始多行 CSV
- 游戏加载后，会把属于同一个 `plotID` 的多行记录组装成一个 `PlotData`

也就是说，JSON 中看到的 `PlotData`，已经是“聚合后的运行时对象”。

```jsonc
{
  "plotName": "试炼开始",
  "spePlot": false,
  "plotID": 123,
  "plotRandomHero": [ { "...": "见 PlotRandomHeroData" } ],
  "differentForce": false,
  "targetHeroID": -1,
  "plotCallFuc": "",
  "randomStartPlot": false,
  "plotDatas": [
    { "...": "见 SinglePlotData" }
  ]
}
```

- `plotName`
  - 剧情名。

- `spePlot`
  - 是否为特殊剧情。

- `plotID`
  - 剧情 ID。
  - 指向 `PlotData` 主库的核心编号。
  - 在 `TheBookOfLong` 的 JSON 补丁层里，这个字段额外支持写成 `"modXXX"` 再映射成实际数字 ID。

- `plotRandomHero`
  - 随机角色选取规则。

- `differentForce`
  - 随机角色是否必须来自不同势力。

- `targetHeroID`
  - 固定目标角色 ID。

- `plotCallFuc`
  - 剧情整体调用的特殊函数。

- `randomStartPlot`
  - 是否随机起始剧情。

- `plotDatas`
  - 单句剧情步骤列表。

### 1.9 SinglePlotData

`SinglePlotData` 对应一条对白、一次镜头切换，或一个可选项节点。

```jsonc
{
  "plotText": "你终于来了。",
  "heroFaceHightLightType": 0,
  "plotSource": 3,
  "sourceName": "1001",
  "plotTarget": 4,
  "targetName": "",
  "choices": [
    { "...": "见 SinglePlotChoiceData" }
  ],
  "clickCallFuc": "",
  "noAutoJump": false,
  "backPic": "bg_room",
  "backBgm": "bgm_story",
  "soundEffect": "Sword",
  "plotShock": false
}
```

- `plotText`
  - 当前对白文本。

- `heroFaceHightLightType`
  - 立绘高亮规则。

- `plotSource`
  - 发言者来源类型。

- `sourceName`
  - 发言者附加参数。
  - 是角色 ID、角色名还是图片名，要看 `plotSource`。

- `plotTarget`
  - 对话对象来源类型。

- `targetName`
  - 对话对象附加参数。

- `choices`
  - 玩家选项列表。

- `clickCallFuc`
  - 点击该句对白后执行的特殊函数。

- `noAutoJump`
  - 是否禁止自动跳到下一句。

- `backPic`
  - 背景图。

- `backBgm`
  - 背景音乐。

- `soundEffect`
  - 音效。

- `plotShock`
  - 是否播放震屏/冲击效果。

## 2. WorldPlotEventData

### 2.1 类的作用

`WorldPlotEventData` 不是对白本体，而是“世界剧情触发规则”。

它所在的位置是：

- `WorldPlotEventController.WorldPlotEventDataBase`

运行流程大致是：

1. `WorldPlotEventController.CheckWorldPlotEventDataBase()` 扫描整张规则表。
2. 先检查时间窗口、重复规则、前置条件。
3. 条件满足后，把规则条目转成 `WorldPlotEventStartData`，挂到当前世界的“待触发剧情事件列表”里。
4. 玩家之后真正做出某种行为，例如进入区域、和某人交互、读书完成、进客栈等，`GameController.CheckPlotTrigger(...)` 才会按 `triggerType + triggerTargetID` 命中。
5. 命中后，才真正把这条规则里的 `plotID` 推进为剧情。

所以它的定位应该理解成：

- “世界剧情候选池”
- “剧情出现规则”
- “剧情什么时候进入可触发状态”

而不是：

- “对白内容”
- “剧情脚本正文”

### 2.2 WorldPlotEventData 主结构

```jsonc
{
  "name": "戴壮寻酒", // 规则名
  "availableMode": 0, // 见 AvailableGameMode
  "difficulty": 0.0, // 主要影响随机事件难度
  "plotID": 231, // 真正要触发的剧情 ID
  "triggerType": 5, // 见 PlotTriggerType
  "triggerTargetID": "戴壮", // 具体触发目标
  "needDatas": [
    {
      "needType": 0, // 见 WorldPlotEventNeedType
      "needTarget": "221",
      "needResult": "1"
    }
  ],
  "startTime": {
    "year": 2,
    "month": 10,
    "day": 15
  },
  "startTimeRandomDayRange": 0, // 起始日期随机浮动天数
  "startContinueTime": 30, // 进入可触发状态后可持续多少天
  "repeatType": 0, // 见 WorldPlotEventRepeatType
  "repeatTime": 0, // 重复间隔天数
  "endTime": {
    "year": 0,
    "month": 0,
    "day": 0
  },
  "startRemindType": 1, // 见 WorldPlotEventStartRemindType
  "startRemindSouce": "姜婉", // 注意：原字段名就是 Souce
  "startRemindText": "魏师兄遭贼人围攻，速回门派救援！",
  "startCallSpeFuc": "PlotGetNewMail;姜婉-……",
  "outtimeCallSpeFuc": "",
  "noAutoDestroy": false,
  "notImportant": false
}
```

### 2.3 字段说明

- `name`
  - 规则名。
  - 复杂 JSON 补丁也是按它做同名覆盖/新增。

- `availableMode`
  - 该规则在哪种游戏模式下有效。

- `difficulty`
  - 规则对应事件的难度。
  - 特别是随机遭遇类触发器，生成的 `EventData` 会继承这项难度。

- `plotID`
  - 真正要触发的剧情 ID。
  - 它指向 `PlotData` 主表中的一条剧情。
  - 在 `TheBookOfLong` 的 JSON 补丁层里，这个字段也支持写成 `"modXXX"` 再映射成数字 ID。

- `triggerType`
  - 玩家最终用什么行为把这条候选剧情真正触发出来。

- `triggerTargetID`
  - 触发目标。
  - 字段含义取决于 `triggerType`：
    - 进入区域时是 `areaID`
    - 进入建筑时常见写法是 `"areaID-buildingID"`
    - 与角色交互时可能直接是角色名
    - 读书/练功完成时可能是书籍或技能 ID

- `needDatas`
  - 规则的前置条件列表。
  - 所有条件都满足后，这条规则才会进入待触发池。

- `startTime`
  - 最早开始生效的日期。

- `startTimeRandomDayRange`
  - 起始日期随机浮动范围。
  - `CheckWorldPlotEventDataBase()` 会基于它把起始日随机后移若干天。

- `startContinueTime`
  - 一旦进入“可触发状态”，最多持续多少天。

- `repeatType`
  - 重复触发规则。

- `repeatTime`
  - 重复间隔天数。

- `endTime`
  - 最晚有效日期。

- `startRemindType`
  - 规则进入可触发状态时，用什么方式提醒玩家。

- `startRemindSouce`
  - 提醒来源。
  - 字段名是原版拼写：`Souce`。
  - 当 `startRemindType = Mail` 时，它通常就是来信署名/发件人。

- `startRemindText`
  - 提醒文本。

- `startCallSpeFuc`
  - 规则激活时额外执行的特殊函数。
  - 从场景数据样例看，常见格式是 `函数名;参数串`。
  - 例如：`PlotGetNewMail;李天临-……-true`

- `outtimeCallSpeFuc`
  - 规则超时失效后执行的特殊函数。

- `noAutoDestroy`
  - 生成的目标事件是否禁止自动销毁。
  - 运行时会把它转换成 `EventData.autoDestroy = !noAutoDestroy`。

- `notImportant`
  - 是否作为“不重要事件”处理。
  - 通常影响 UI 呈现、提示权重或事件筛选。

### 2.4 WorldPlotEventNeedData

`WorldPlotEventNeedData` 表示一条前置条件。

```jsonc
{
  "needType": 0,
  "needTarget": "221",
  "needResult": "1"
}
```

- `needType`
  - 条件类型。

- `needTarget`
  - 条件目标。
  - 是剧情 ID、日志键名、角色名、任务 ID，要看 `needType`。

- `needResult`
  - 条件结果。
  - 是 `0` / `1`、`alive` / `dead` / `sameforce`，还是 `>3`、`<10` 这种比较表达式，也取决于 `needType`。

### 2.5 WorldPlotEventData 相关枚举

#### AvailableGameMode

- `0 = Plot`
  - 剧情模式。
- `1 = Free`
  - 自由模式。
- `2 = Both`
  - 两种模式都可用。

#### PlotTriggerType

- `0 = None`
  - 无具体触发行为。
- `1 = BigMapRandomEvent`
  - 大地图随机事件。
- `2 = AreaRandomEvent`
  - 区域随机事件。
- `3 = EnterArea`
  - 进入区域。
- `4 = EnterBuilding`
  - 进入建筑。
- `5 = InteractHero`
  - 与角色互动。
- `6 = FinishReadBook`
  - 读书完成。
- `7 = FinishStudySkill`
  - 练功/学技能完成。
- `8 = StayInArea`
  - 在区域内停留。
- `9 = EnterInn`
  - 进入客栈。

#### WorldPlotEventNeedType

- `0 = PlotHappened`
  - 某个剧情是否已经发生。
  - 常见写法：`needTarget = 剧情ID`，`needResult = 1/0`。

- `1 = PlotEventLog`
  - 读取剧情事件日志。
  - 常见写法：
    - `needTarget = 日志键`
    - `needResult = 0`
    - `needResult = 1`
    - `needResult = >3`
    - `needResult = <10`
    - `needResult = 5`

- `2 = HeroState`
  - 检查某个角色状态。
  - 从反编译逻辑已确认常见值包括：
    - `alive`
    - `dead`
    - `sameforce`

- `3 = MissionFinished`
  - 某个任务是否已完成。
  - 常见写法：`needTarget = 任务ID`，`needResult = 1/0`。

#### WorldPlotEventRepeatType

- `0 = NoRepeat`
  - 不重复。

- `1 = NoPlotTriggerRepeat`
  - 名字直译是“未触发剧情前可重复、剧情真正触发后不重复”这类控制。（按命名和调用逻辑推断）

- `2 = Repeat`
  - 允许重复。

#### WorldPlotEventStartRemindType

- `0 = NoRemind`
  - 不提醒。
- `1 = Mail`
  - 以邮件/来信形式提醒。
- `2 = WorldEvent`
  - 以世界事件/江湖传闻形式提醒。

### 2.6 与 PlotData 的关系

`WorldPlotEventData` 不直接存对白内容，只存一个 `plotID`。  
真正的剧情文本、角色立绘、选项、音效、背景等内容，仍然在 `PlotData` 中。

因此可以这样理解：

```text
WorldPlotEventData
  负责：什么时候能触发某段剧情

PlotData
  负责：这段剧情具体演什么
```

### 3.3 最容易混淆的点

#### `MissionData` 不是“任务存档实例”

它更接近“任务原型”。  
真正被玩家接到手里的任务，是从它 `Clone()` 出来的实例。

#### `WorldPlotEventData` 不是“剧情文本”

它只是决定某条剧情什么时候进入可触发状态。  
对白正文仍然在 `PlotData`。

#### `plotID` 才是剧情桥梁

很多系统最后都会落到 `plotID`：

- `WorldPlotEventData.plotID`
- `PlotData.plotID`
- `MissionTargetData.missionTargetFinishCallPlotID`

其中：

- `WorldPlotEventData.plotID` 直接决定“这条世界剧情规则最后触发哪段剧情”
- `PlotData.plotID` 是剧情对象自己的编号
- `missionTargetFinishCallPlotID` 则是“任务阶段完成后追加一段剧情”
